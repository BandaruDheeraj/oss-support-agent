## 2) What competitors do (why easier)

- **Competitor mechanism:** Braintrust exposes `init()` and optional `initLogger()` in one SDK.
- **Why easier:** Bootstrap path is shorter and avoids direct OTel pipeline wiring for initial setup.

## 3) What we do today in oss-support-agent

- **Current behavior:** Arize adapter manually creates exporters, span processors, and provider resources before any eval logging.
- **Affected components:** evals/platforms/arize.ts, core/observability/arize.ts, package.json

## 4) Gap statement

Because Arize adapter manually creates exporters, span processors, and provider resources before any eval logging., users cannot Get first trace/eval from JS quickly as easily as with Braintrust exposes `init()` and optional `initLogger()` in one SDK..

## 5) Proposed fix

- **Owner-scoped change:** Ship `@arizeai/ax-js` bootstrap helper (`initTracing`) to configure exporter/provider/resources in one call.
- **Root cause:** No high-level Arize JS bootstrap that hides low-level OTel plumbing.
