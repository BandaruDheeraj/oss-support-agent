## 2) What competitors do (why easier)

- **Competitor mechanism:** Braintrust `Eval`/row logging keeps score payloads native to the SDK data model.
- **Why easier:** Teams can attach scores without inventing key names per project.

## 3) What we do today in oss-support-agent

- **Current behavior:** Arize/OpenInference adapter writes custom `evaluation.key.*` fields for row and aggregate metrics.
- **Affected components:** evals/platforms/arize.ts, evals/platforms/braintrust.ts, evals/run-eval.ts

## 4) Gap statement

Because Arize/OpenInference adapter writes custom `evaluation.key.*` fields for row and aggregate metrics., users cannot Standardize eval-row and summary scoring semantics as easily as with Braintrust `Eval`/row logging keeps score payloads native to the SDK data model..

## 5) Proposed fix

- **Owner-scoped change:** Add OpenInference eval semantic conventions and helper mappers for row + run summary metrics.
- **Root cause:** OpenInference semantic conventions do not define a canonical score payload profile for benchmark rows.
